# PLAN-fifa-details-sync-pipeline — generate match-details.json, validate, ship via the existing workflow

> **GATE: Brief 2 starts only after explicit map approval (Brief 1 complete).**

## Rank and leverage

**Rank 2 of 6 (Brief 2).** Turns the frozen contract (Plan 1) into real generated
data on the same three-times-daily cadence the results already use. Once this lands,
browser work (Plans 3–5) has genuine data to render, and the validator gate keeps a
malformed feed from ever shipping.

## Goal and user-visible outcome

`docs/data/match-details.json` exists, is regenerated by
`.github/workflows/sync-results.yml` alongside `results.json`, is validated before
commit, and contains a `complete` or `partial` record for every completed bracket
match (M73–M97 today; grows as games finish). Still no UI change.

## In scope

- Extend `scripts/fetch_results.py` minimally: **preserve FIFA source IDs**
  (idCompetition/idSeason/idStage/idMatch) through `match_all` instead of
  discarding them, and call the new detail-cache update at the end of a successful
  sync.
- Add the orchestration entry point in `scripts/match_details.py` (`main()` /
  `update_details(topology, results, old_details, calendar_feed, fetch_fn)`):
  identify completed matches with missing/partial/score-mismatched records → fetch
  per-match payloads only for those → parse conservatively → merge → deterministic
  write.
- Failure behavior: per-match failure ⇒ validated `partial` record + visible
  warning naming bracket code and source match ID; retried next run. Malformed
  top-level contract ⇒ validator fails ⇒ workflow does not commit.
- Workflow update: run sync → run `scripts/validate_match_details.py` → commit
  `results.json` + `match-details.json` together when either changed → single
  Pages trigger → keep the existing failure-issue path.
- A `--dry-run` flag mirroring `fetch_results.py`'s.

## Out of scope

- Browser loading/rendering (Plans 3–4), portrait mapping data (Plan 5).
- FotMob (blocked branch — see appendix), odds (banned).
- Any change to what `results.json` itself contains, or to the sync schedule.

6. Full sweep: Python tests + all JS tests green; served dashboard unaffected.

## Data contracts and invariants

- `results.json` remains byte-compatible with today's schema; details live only in
  `match-details.json`.
- A newly completed score is never blocked by a detail-fetch failure.
- The validator runs before any workflow commit; a contract violation stops the
  ship, visibly.
- Deterministic serialization (sorted keys, fixed separators, trailing newline
  matching `fetch_results.py`'s style).

## Edge cases and codebase traps

- **Do not re-fetch everything each run** — only missing/partial/score-changed
  records (the merge rules from Plan 1 encode this). 25+ per-match fetches ×3/day
  would be rude to the feed and slow the job.
- **`enrich_fifa_goals` already hits the live endpoint for recent games** — reuse
  its fetched payloads where possible in the same run instead of fetching twice
  for the same match.
- The workflow's commit step currently greps only `results.json` in porcelain —
  extend the pattern, don't duplicate the step (two commits would double Pages
  builds).
- Local generation writes into `docs/data/` — that's the deployed tree; fine for
  the sandbox working tree, but never commit here (human gate).
- Time zones: kickoff stays the FIFA ISO string; do not localize in data.
