my-wc26-bracket — analysis & roadmap

What this codebase is, what three expert lenses found, the 5 plans that came out of it, and the lessons behind each decision. Generated July 7, 2026 — mid-tournament (R16 ends today, QFs start July 9).

What this thing is

A zero-backend World Cup bracket dashboard. A user drops in their Excel bracket (or clicks winners in a builder); the browser parses it, scores it against live results, and renders a full dashboard. A GitHub Action fetches real results from FIFA 3×/day and commits them to one JSON file. GitHub Pages serves everything static. ~3,100 lines total, no build step, no npm dependencies.

🧒 ELI5: Imagine a scoreboard poster that redraws itself. You hand it your list of guesses once. A robot mailman drops off a note with the real game scores three times a day. Your own web browser does all the math and drawing — nothing about your guesses ever leaves your computer.

🧒 ELI5 — what's a "pure function" and why care? render.js is like a vending machine: same coins in → same snack out, every time, no surprises. That's why you can test it by just comparing its output text to a saved copy. If it secretly checked the time or the weather, testing would be much harder.

What's genuinely good here (worth protecting)

✅ The scoring property test (tests/scoring.mjs) builds an independent scorer from first principles and checks the engine against it on 3,000 random brackets, plus the invariant that points always sum to 80. This is the strongest test in the repo.
✅ The sync is defensive: dual sources with auto-fallback, never clobbers a decided result, dedupes the "two teams met twice" case by date, guards shootout-score mangling, opens a GitHub issue on failure.
✅ No build step: cd docs && python3 -m http.server is the whole dev loop.

The three problems that matter most

⚠️ 1. The tests are decoration. No CI runs them. The golden test needs /tmp/py_sections.json (from another repo's Python), the parse test needs a private .xlsx at an absolute path on this Mac. A bad commit to render.js ships to every visitor unchecked — during a live tournament.
⚠️ 2. results.json is a single point of failure with no gate. The bot commits whatever the pipeline produces. And the scariest failure isn't a crash — it's silence: if FIFA renames a team and team_map.json lacks the alias, that match simply never resolves. No error, no issue, the bracket just stops updating.
⚠️ 3. Freshness is oversold. The page says "Live · updated …" but syncs run at 3 fixed times a day. On July 9 (QF day), a 2 PM result can sit hidden until the 6 PM sync — and the UI gives no hint it's stale.

Three expert lenses

I first drafted a top-5 from raw code reading, then re-examined it through how three well-known engineers think. Each lens killed or promoted items. (These are my applications of their public philosophies — not quotes.)

🧪 Andrej Karpathy — simplicity, data quality, tests that actually run

"A test you don't run is a comment. The data pipeline IS the product. Keep the whole thing small enough to hold in your head."

  • Would praise: no-build vanilla JS, the pure render function, the property-based scoring test (3,000 random brackets — very his style).
  • Would call out: the golden test's dependency on a Python dump in /tmp — the parity project is done; freeze the output as a repo-owned snapshot and let the JS engine own its future. And validate the synced data — garbage in results.json means a wrong scoreboard for every visitor, which no amount of UI polish fixes.
  • Effect on the list: promoted the CI/hermetic-tests work from "chore" to #1; killed a half-baked "rewrite render.js into components" idea (churn, no user value, breaks the parity guarantee for nothing).

🔗 Scott Hanselman — the URL is the app; sharing is the feature; a11y is table stakes

"The most powerful database is the URL. If your users are in a pool together, the product is the *between-people* part. And if it doesn't work with a keyboard, it doesn't work."

  • Would praise: privacy story ("your file never leaves the browser") told honestly; focus-visible styles and reduced-motion already in the CSS.
  • Would call out: a bracket is 31 bits — why is sharing it a JSON-file-and-instructions ritual? Put it in the URL hash. The pool is social but the app is single-player: no compare, no standings between Rob's manual updates. Stat cards are hover-only — keyboard and touch users get nothing.
  • Effect on the list: created plan #3 (share links) and #4 (compare/leaderboard) as distinct, sequenced pieces; folded a11y into a concrete hardening plan instead of a vague "improve accessibility" item.

🛡️ Mark Russinovich — failure modes, blast radius, silent failure is the enemy

"Ask: what's the blast radius when this fails? Who notices, and when? The failures that hurt aren't the crashes — they're the ones that produce nothing."

  • Would praise: idempotent sync, issue-on-failure step, dual-source fallback, deterministic JSON writes for minimal diffs.
  • Would call out: the commit path has no schema gate (validate before publish, not after users see it); the team-name-mapping gap fails by absence — build a detector that notices a finished knockout game that matched nothing; increasing cron frequency naively would create 26 no-op commits+deploys a day (observability noise that buries signal).
  • Effect on the list: added the validator gate to plan #1, and the stall detector + commit-throttle to plan #2. Also the insight that the freshness chip must live outside the golden-tested render path.
🧒 ELI5 — why three lenses? It's like asking a chef, a waiter, and a fire marshal to walk through the same restaurant. The chef checks the kitchen (code & data quality), the waiter checks the guest experience (sharing, accessibility), the fire marshal checks what burns down and who smells the smoke (failure modes). Same room, three different dangers spotted.

The 5 plans, ranked by leverage

Leverage = (value to users × risk removed) ÷ effort, weighted by the tournament clock. Each plan is a standalone PLAN-*.md in the repo root, written so a less capable model can execute it without questions.

1

PLAN-ci-safety-net.md — hermetic tests + gated data pipeline

Effort: ~half a day · Files: tests/*, scripts/validate_results.py, 2 workflows · do first

Freeze the golden test into a repo-owned snapshot (with frozen inputs — the live results.json churns 3×/day and would break a naive snapshot). Make the private-workbook test skip cleanly. Run everything in Actions. Add a stdlib Python validator that must pass before the sync bot commits.

Why #1 — It de-risks every other plan (three of them touch the render/glue path) and it's the only one that protects the site this week against a bad feed or a bad commit. Insurance is cheapest before the storm: QFs start July 9.

🧒 ELI5: Right now the smoke detectors are in a drawer. Plan 1 screws them to the ceiling (CI) and adds a taste-tester who checks the robot mailman's letters before they're pinned to the fridge (the validator).
2

PLAN-match-day-freshness.md — denser sync, honest staleness, stall detection

Effort: ~half a day · Files: fetch_results.py, sync workflow, index.html, main.js · time-sensitive (Jul 9)

Cron every 30 min in the match window; a truthful "results updated 23m ago" chip (computed client-side from a new refreshed_iso, living in the viewer bar — not in the golden-locked render output); throttle stamp-only commits; detect finished knockout games that matched no bracket fixture (the silent team_map gap).

Why #2 — during the QFs "is this current?" is the entire user experience, and the silent-stall detector covers the one failure mode that today would never be noticed by anyone.

🧒 ELI5: The robot mailman now comes every half hour on game days, the fridge note gets a timestamp sticker, and there's an alarm for the case where a letter arrives addressed to a name nobody recognizes (so it would otherwise go in the bin unnoticed).
3

PLAN-share-links.md — your whole bracket in a URL

Effort: ~1 day · Files: NEW share.js, main.js, index.html, NEW test · biggest product win

A bracket is exactly 31 binary choices + name + tiebreaker → ~80 chars of base64 in the URL hash. "Share link" button; opening a link renders the sender's dashboard view-only (never clobbering the visitor's own saved bracket). Zero backend; the privacy promise holds.

Why #3 — highest joy-per-line-of-code in the repo, and #4 depends on its decoder. After freshness, because a wrong-but-shareable scoreboard is worse than an unshareable right one.

🧒 ELI5: Your 31 guesses are just 31 coin flips — heads or tails per game. That's so little information it fits inside the web address itself, like writing your locker combination on the invite instead of mailing the whole locker.
4

PLAN-compare-brackets.md — client-side mini-leaderboard

Effort: ~1 day · Files: NEW compare.js, main.js, index.html, NEW test · depends on #3

Import colleagues' brackets (JSON or share-link), rank everyone by confirmed points (reusing the exported pure computeState), and show a "where we differ" list for the games still to play, sorted by points at stake.

Why #4 — the pool's actual social loop ("am I beating Dave?") with no server. Ranked below #3 only because it builds on it and touches more UI surface.

🧒 ELI5: Everyone brings their guess-sheet to your kitchen table and you line them up side by side. Nothing leaves the kitchen — your browser does the ranking.
5

PLAN-hardening-a11y.md — five small fixes that close real holes

Effort: ~half a day, five independent commits · Files: render.js, main.js, storage.js, interact.js

(1) escape the 4 unescaped note interpolations — byte-neutral today, which is exactly why it's safe; (2) stop the demo button from silently wiping your what-if edits (a real data-loss bug); (3) stat cards on keyboard focus/touch; (4) r.ok checks + offline cache fallback; (5) aria-live on the search count.

Why #5 — real but small; none blocks the others; each is a clean single-commit fix.

🧒 ELI5: Tightening five loose screws you found while walking the house: none is an emergency, but one of them (the demo wipe) is quietly eating people's homework.

Step 3 Run what can be run

Ran scoring.mjs (passes — 3,001 brackets) and builder.mjs (passes). Confirmed the two external test dependencies exist on this machine only. 🧒 ELI5: Don't just read the recipe — taste the soup. Two tests passed on my stove; two only work in the author's kitchen. That gap is finding #1.
Step 4 Apply the three lenses, then re-rank

Each lens is a question you ask the same facts: K: "does the feedback loop actually run? is the data trustworthy?" H: "where's the human/social/URL leverage?" R: "what fails silently, and what's the blast radius?" The final ranking is where the lenses agree: all three point at the unguarded results.json path first.

Step 5 Write plans for a weaker executor

Each plan states goal → exact files → ordered steps → the traps → verifiable acceptance criteria. The traps section is the point: a capable model discovers "the snapshot must freeze its inputs because a bot rewrites results.json 3×/day" by itself, painfully, after CI goes red; a plan that says it up front makes the weaker model succeed on the first pass. Acceptance criteria are all commands or observable behaviors — no "should work correctly".

🧒 ELI5: Good instructions for a new helper aren't just "build the shelf" — they're "the third screw strips easily, turn it gently." The traps are the whole value of experience.

Learning moments 🎓

Six transferable ideas from this codebase — each with the concrete instance here, the general principle, and an ELI5.

1 · A test you don't run is a comment

Here: four decent test files, zero CI. The golden test even requires dumping a file from a different repo's Python — so it silently stopped being run the moment porting finished.

Principle: a test's value is its execution frequency × its ability to fail. Hermetic (no outside-the-repo inputs) is what makes frequent execution possible. Whenever you see a test with an absolute path in it, you've found a test that will quietly die.

🧒 A fire drill you never run is just a poster on the wall. And a drill that requires borrowing the neighbor's keys will never be run.
2 · Snapshot tests must freeze their inputs, not just their outputs

Here: the obvious "commit the golden output as a fixture" fix has a booby trap: the render input (results.json) is rewritten by a bot 3×/day. Snapshot the output alone and CI turns red at 6 PM every day.

Principle: a snapshot is a statement "f(frozen input) = frozen output". If any input keeps moving, you haven't frozen a function — you've frozen a moment. This is the single most common way snapshot testing fails in the wild (timestamps, locales, live data).

🧒 A photo of your sandcastle only proves it's unchanged if the tide isn't coming in. Photograph the castle and bottle the tide.
3 · The deadliest failure is silence (absence, not error)

Here: if FIFA's feed spells a team differently than team_map.json expects, the match result simply never attaches to the bracket. No exception, no failed workflow, no issue. The dashboard just… stops moving, and looks exactly like "no games played yet".

Principle: error handling catches things that happen; you also need detectors for things that should have happened and didn't. The trick is finding an observable proxy — here: "a finished knockout game in the feed that matched no bracket fixture" is loud, cheap, and specific. (And it needs a whitelist: the third-place playoff legitimately matches nothing!)

🧒 A smoke alarm tells you about fire. Nothing tells you the mailman stopped coming — unless you write down "mail should arrive daily" and check. Silence needs its own alarm.
4 · Validate at the boundary you control, before publishing

Here: the sync bot commits results.json straight to the branch that Pages serves. The validator goes between fetch and commit: a bad feed then fails the workflow (which already opens an issue) and visitors keep yesterday's good data instead of today's garbage.

Principle: every pipeline has one moment where data crosses from "mine, checkable" to "published, everyone's problem". Gate that moment. Post-hoc monitoring tells you users were hurt; a gate means they weren't.

🧒 Taste the soup in the kitchen, not by watching the customers' faces.
5 · Know which of your outputs are byte-locked — and route around them

Here: render.js output is protected by a byte-exact snapshot. So the freshness chip goes in the viewer bar (main.js territory), the aria-live attribute is set on the DOM after injection, and the new JSON key is one render.js never reads. Meanwhile escaping note IS safe inside the locked zone — because for all current values it's byte-neutral, and the passing snapshot proves it.

Principle: strong guarantees (byte parity, API contracts, wire formats) are walls. Good changes either go around the wall, or demonstrably don't touch it. Knowing which is which is most of the skill.

🧒 There's a "fresh paint" wall in the house. You can hang things on every other wall, and you can touch the painted one only if your hands are provably clean — the paint check (the test) tells you.
6 · Count the bits before you build the feature

Here: "share your bracket" sounds like accounts, a database, uploads. But a bracket is 31 binary choices — under 4 bytes of information. That fits in a URL with room to spare, so sharing needs zero infrastructure and the privacy promise survives intact. The same counting killed fancier encodings: base64-of-JSON is 80 chars and debuggable; bit-packing saves 40 chars nobody needs.

Principle: before designing a system, measure the information actually flowing. Tiny state → the URL is your database. It's the highest-leverage sizing question in product engineering.

🧒 You don't rent a moving truck for a house key. Count what you're actually carrying first — sometimes it fits in your pocket (the web address).
Bonus · The demo-wipe bug: convenience features can eat user data

Here: the what-if overrides are namespaced per-bracket by hash (smart!), but "view the demo" runs the same reset path as "upload a new bracket" — so a preview destroys real edits. The plan fixes it with a stash/restore rather than skipping the reset, because the reset is still correct for genuine replacements.

Principle: whenever a destructive cleanup is triggered by state-change detection, list every path that changes the state. Previews, demos, and "just looking" flows are the classic false positives.

🧒 The robot vacuum is great — until it eats the LEGO castle you left on the floor "just for a minute". Teach it the difference between mess and a work-in-progress.
